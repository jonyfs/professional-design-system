# Feature Specification: Curated Theme Presets

**Feature Branch**: `017-curated-theme-presets`

**Created**: 2026-07-12

**Status**: Draft

**Input**: User description: "este design system deve ter no mínimo 40 temas
já pre existentes para que o usuário possa escolher. Busque ideias na
internet sobre estes temas e adicione isso ao design system dando opcões
profissionais para que ao usar seja percebido que é uma ideia de ótima
qualidade, polido, perfeito, pensado com /ui-ux-pro-max /frontend-skill
/frontend-design:frontend-design . Use temas de bom gosto, deixando o
visual profissional e sem parecer que foi algo criado por IA"

(Translation: this design system must ship at least 40 pre-built themes
for users to choose from — researched from real-world references,
professional-grade, tasteful, polished, and deliberately avoiding a
generic/AI-generated look.)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Browse and preview the theme collection (Priority: P1)

A user evaluating this design system for a new product wants to quickly
see the range of available visual identities before committing to one,
comparing how the same components (buttons, cards, badges, forms) look
under different palettes without writing any code.

**Why this priority**: Without a way to browse and compare themes side by
side, the other 39+ themes are invisible inventory — this is the entry
point that makes the whole feature discoverable and demonstrates the
"professional, tasteful, not AI-generated" quality bar the request cares
about most.

**Independent Test**: Open the theme gallery page, confirm at least 40
distinct named themes are listed, each with a live preview of representative
components (not just color swatches), and confirm switching the preview
does not reload the page or lose scroll position.

**Acceptance Scenarios**:

1. **Given** the theme gallery page, **When** it loads, **Then** at least
   40 themes are listed, each identified by a distinct, real name (not
   "Theme 1", "Theme 2").
2. **Given** a theme in the gallery, **When** the user selects it,
   **Then** a representative set of already-shipped components (e.g.
   Button, Badge, Card, Alert) re-renders live using that theme's palette.
3. **Given** two different themes previewed in sequence, **When** compared,
   **Then** each is visually distinct enough that a user could tell them
   apart from a screenshot alone (no two themes are near-duplicates of
   each other).

---

### User Story 2 - Apply a chosen theme across the whole design system (Priority: P1)

A user who has picked a theme wants every component in the catalog — not
just the ones shown in the preview — to consistently reflect that theme's
palette, so their product looks cohesive rather than having some screens
themed and others not.

**Why this priority**: Equal priority to Story 1 — a theme that only
covers a handful of "preview" components isn't usable in a real product;
this is what makes the feature actually deliver on "choose a theme," not
just "look at some color swatches."

**Independent Test**: Apply a non-default theme, then visit every
existing component's own demo page and confirm each one reflects the
chosen theme's palette rather than the original default colors.

**Acceptance Scenarios**:

1. **Given** a theme has been selected, **When** the user navigates to any
   previously-shipped component's page, **Then** that component renders
   using the selected theme's palette, not the original default.
2. **Given** a theme has been selected, **When** the user reloads the page
   or returns in a new session, **Then** the same theme is still applied
   (the choice persists without requiring the user to reselect it).
3. **Given** any theme is active, **When** any component's text or
   interactive element is inspected, **Then** it still meets this design
   system's existing accessibility contrast requirements — a theme MUST
   NOT be allowed to ship if it degrades accessibility below the
   already-ratified bar.

---

### User Story 3 - Trust that every theme is genuinely production-quality (Priority: P2)

A user choosing a theme for a real, client-facing product wants
confidence that whichever of the 40+ options they pick will look
deliberate and professional — not that some are polished flagship
options and others are filler added just to hit a quota.

**Why this priority**: Directly answers the request's explicit quality bar
("polido, perfeito," "sem parecer que foi algo criado por IA"). Lower
priority than Stories 1-2 only because it's a quality gate on the same
deliverable, not a separate user-facing capability — it shapes HOW the 40
are built, not whether the feature exists.

**Independent Test**: For a random sample of themes across the collection,
confirm each has a distinct rationale/inspiration documented (not
auto-generated by algorithmically rotating a hue value) and confirm no
two themes in the full set share a near-identical palette.

**Acceptance Scenarios**:

1. **Given** the full theme collection, **When** any individual theme is
   examined, **Then** it has an intentional name and a real design
   rationale (e.g. inspired by a recognizable aesthetic tradition,
   material, or well-established color-scheme convention), not a
   procedurally-generated label like "Blue 3."
2. **Given** the full theme collection, **When** scanned as a set,
   **Then** no two themes are near-duplicate palettes (same hue family and
   same neutral/accent structure with only a marginal shift).

---

### Edge Cases

- What happens when a user's browser has no persisted theme preference
  (first visit)? The design system MUST fall back to one specific,
  clearly-designated default theme, never a random one.
- What happens if a theme is removed or renamed in a future update while
  a user has it persisted as their choice? The system MUST fail
  gracefully back to the default theme rather than rendering with no
  theme applied at all.
- What happens for a user with an OS/browser-level "prefers reduced
  color"/high-contrast accessibility setting enabled? Every theme MUST
  still independently satisfy this design system's existing WCAG
  contrast requirements — no theme is exempted from the accessibility bar
  as a tradeoff for visual distinctiveness.
- What happens when two themes are visually similar by coincidence (e.g.
  two blue-centric options)? Each MUST still be distinguishable via a
  clear name and a genuinely distinct supporting palette (differing
  neutrals, accents, or overall mood), not merely a hue rotation of the
  same structure.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST provide at least 40 distinct, named, pre-built
  themes for users to choose from.
- **FR-002**: Each theme MUST define a complete palette (not just a single
  accent color swap) covering every semantic role this design system's
  existing token structure already defines (brand/primary, neutral scale,
  success, warning, error, info).
- **FR-003**: System MUST provide a browsable gallery where all themes are
  visible and previewable using real, representative components, not
  color swatches alone.
- **FR-004**: Selecting a theme MUST apply it consistently across every
  previously-shipped component in the catalog, not a subset.
- **FR-005**: A user's theme selection MUST persist across page reloads
  and future visits without requiring reselection.
- **FR-006**: System MUST fall back to one specific, clearly-designated
  default theme whenever no valid persisted selection exists.
- **FR-007**: Every theme MUST independently meet this design system's
  existing accessibility bar (WCAG AAA text contrast / WCAG AA non-text
  contrast for every color pairing it introduces) — no theme ships
  without this verification, regardless of how visually appealing it is.
- **FR-008**: Each theme MUST have a distinct, intentional name and a
  documented design rationale (its inspiration or reference point), not a
  procedurally-labeled placeholder.
- **FR-009**: No two themes in the collection MUST be near-duplicate
  palettes (same hue family and neutral/accent structure with only a
  marginal shift) — each MUST be genuinely, visibly distinct from every
  other.
- **FR-010**: System MUST NOT introduce a theme that requires a different
  component markup structure than already exists — a theme changes
  color/typography/surface values only, never a component's HTML
  structure or behavior.

### Key Entities

- **Theme**: a named, complete palette mapping this design system's
  existing semantic color roles (brand, neutral scale, success, warning,
  error, info) to a specific, intentional set of values, plus a short
  design rationale/inspiration note.
- **Theme Collection**: the full set of 40+ themes, browsable and
  comparable as a group.
- **User's Theme Selection**: a single persisted choice of which theme is
  currently active, defaulting to one designated theme when unset or
  invalid.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: At least 40 distinct, named themes are available for
  selection.
- **SC-002**: 100% of previously-shipped components render correctly and
  consistently under every theme in the collection (no component is
  left un-themed or visually broken under any theme).
- **SC-003**: Every theme independently passes this project's automated
  accessibility/contrast audit at the same bar already required for the
  default theme — zero exceptions.
- **SC-004**: A user can identify and switch to a specific theme from the
  gallery in under 30 seconds without any documentation.
- **SC-005**: In an informal side-by-side review, no two themes in the
  collection are mistaken for each other (each has a clearly distinct
  visual identity).
- **SC-006**: A user's selected theme remains active across a full
  session restart (browser close/reopen) with no reselection needed.

## Assumptions

- "Themes" means complete, self-contained color palettes swapped in for
  this design system's existing semantic token roles (brand, neutral
  scale, success/warning/error/info) — not per-component visual redesigns,
  not new component variants, and not a change to any component's markup
  or interaction behavior. This mirrors the established convention used
  by comparable curated-theme systems researched for this feature
  (DaisyUI's 30+ named themes, Bootswatch/Bootstrap Dynamic Themes'
  40-50+ palette collections) — swap the palette, keep the component
  contract identical.
- Dark-toned and light-toned palettes are both represented directly within
  the 40+ themes themselves (some themes are inherently dark, others
  inherently light), rather than layering a separate universal light/dark
  mode toggle on top of every theme — this matches the researched
  precedent (DaisyUI ships explicitly light themes like "light"/"cupcake"
  alongside explicitly dark ones like "dracula"/"night" as peers in one
  list, not as a 2x multiplier applied to every theme) and avoids
  doubling the required set to 80+ combinations.
- Theme selection is a client-side preference (persisted locally to the
  browser, e.g. via `localStorage`), not a server-side/account-level
  setting — no backend/authentication system exists in this project to
  attach a preference to.
- Given the explicit request to avoid a "criado por IA" (AI-generated)
  look, each theme's palette and name draw on well-established,
  professionally-recognized color-scheme traditions and references
  (rather than procedurally-generated hues) — the specific reference
  points are a research task for the planning phase, not a business
  requirement to enumerate here.
- Given the scale of this request (40+ fully-verified, distinct,
  professional themes — each independently re-clearing this project's
  existing WCAG AAA/AA bar across all ~48 previously-shipped components),
  delivery may reasonably be planned/executed in batches within this
  feature rather than as 40 simultaneous, hand-verified palettes in one
  pass — this is an implementation-sequencing detail for the planning
  phase, not a change to the 40-theme requirement itself.
- This feature ships as an addition to the existing static HTML + Tailwind
  system, consistent with every prior feature's technology choice; a
  React-package equivalent is deferred to a future React port batch,
  mirroring how static features have historically preceded their React
  ports in this project.
